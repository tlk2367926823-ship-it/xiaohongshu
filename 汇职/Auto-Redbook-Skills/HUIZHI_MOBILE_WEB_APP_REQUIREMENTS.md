# 深圳汇职驾校小红书分享活动 Mobile Web App 需求文档

## 1. 项目背景

基于现有 `Auto-Redbook-Skills` 小红书内容生成能力，开发一个面向深圳汇职驾校线下获客场景的移动端网页。用户通过扫描二维码或 NFC 碰一碰进入页面，以尽可能少的操作完成“小红书分享内容生成、素材匹配、发布引导、奖励确认”的闭环。

本项目定位为接近正式产品体验的前端 MVP，不开发后端、数据库、登录系统，也不直接实现真实小红书自动发布。

## 2. 产品目标

- 让用户从进入页面到生成分享内容控制在 1 次核心点击内完成。
- 自动生成适合小红书发布的标题、正文和标签，降低用户编辑成本。
- 自动匹配本地 Mock 图片或视频素材，形成完整分享草稿。
- 预留发布模块扩展点，后续可接入真实小红书发布、复制跳转、截图分享或服务端发布能力。
- 在发布成功后展示奖励页面，形成清晰的活动闭环。

## 3. 目标用户与使用场景

目标用户：

- 到店咨询或训练中的驾校学员。
- 通过朋友推荐了解深圳汇职驾校的潜在学员。
- 线下活动现场扫码参与分享的用户。

典型场景：

- 门店前台、训练场、教练车、活动海报放置二维码或 NFC 标签。
- 用户扫码或碰一碰后进入活动页。
- 用户点击「立即发布」，系统自动生成分享内容和素材。
- 用户确认内容后进入发布引导流程。
- 用户完成发布后回到页面领取奖励。

## 4. MVP 范围

### 包含

- 移动端单页 Web App。
- 活动介绍页。
- 通义千问 API 前端调用。
- API Key 通过环境变量配置。
- 小红书标题、正文、标签生成。
- 本地 Mock 图片/视频素材匹配。
- 生成结果预览页。
- 发布流程引导页。
- 发布成功后的奖励页。
- 发布模块抽象设计，便于后续替换为真实发布能力。
- 基础错误处理、加载状态、重试机制。

### 不包含

- 后端服务。
- 数据库。
- 用户登录、手机号验证、账号体系。
- 真实小红书自动发布。
- 真实奖励核销系统。
- 服务端 API Key 保护。

## 5. 核心用户流程

1. 用户通过二维码或 NFC 进入活动页面。
2. 页面自动识别来源参数，例如 `?source=qrcode`、`?source=nfc`、`?campaign=summer_driving`。
3. 用户查看活动介绍、奖励说明和分享预览。
4. 用户点击「立即发布」。
5. 前端调用通义千问 API，根据预设 Prompt 生成：
   - 小红书标题
   - 正文
   - 标签
   - 推荐素材类型
   - 推荐发布风格
6. 系统根据生成结果和活动规则，从本地 Mock 素材中匹配图片或视频。
7. 展示生成结果预览，用户可一键复制文案、切换素材、重新生成。
8. 用户点击「去发布」进入发布引导。
9. MVP 阶段采用模拟发布流程：
   - 复制文案
   - 保存或展示素材
   - 引导用户打开小红书手动发布
   - 用户点击「我已发布」
10. 页面进入奖励页，展示领取方式和活动提示。

## 6. 页面结构

### 6.1 活动首页

目标：快速说明活动价值，并推动用户点击「立即发布」。

内容：

- 品牌：深圳汇职驾校
- 主标题：例如「分享学车体验，领取专属奖励」
- 简短利益点：例如「AI 自动生成小红书文案，1 分钟完成分享」
- 活动奖励说明
- 本地素材预览
- 主按钮：「立即发布」

交互：

- 页面加载后展示轻量动效。
- 主按钮固定在底部安全区，方便拇指点击。
- 如果 URL 带有 campaign/source 参数，页面可展示对应活动文案。

### 6.2 内容生成页

目标：让用户感知系统正在自动完成复杂工作。

内容：

- 生成进度状态：
  - 正在分析活动信息
  - 正在生成小红书标题
  - 正在匹配图片/视频素材
  - 正在整理发布草稿
- 动态加载动画。

交互：

- 点击「立即发布」后自动进入此页。
- 成功后自动跳转到生成结果页。
- 失败后提供「重新生成」按钮。

### 6.3 生成结果页

目标：展示可发布草稿，并让用户尽快进入发布流程。

内容：

- 标题预览。
- 正文预览。
- 标签预览。
- 匹配的图片或视频素材。
- 操作按钮：
  - 「去发布」
  - 「复制文案」
  - 「换一组」
  - 「重新生成」

交互：

- 默认展示系统推荐版本。
- 「换一组」只切换本地 Mock 素材，不重新调用 API。
- 「重新生成」重新调用通义千问 API。
- 文案较长时使用折叠预览，避免页面压迫感。

### 6.4 发布引导页

目标：在不接入真实小红书发布的前提下，让用户顺畅完成手动发布。

内容：

- 发布步骤：
  - 文案已准备好
  - 保存/查看素材
  - 打开小红书发布
  - 发布完成后返回领取奖励
- 按钮：
  - 「复制文案」
  - 「查看素材」
  - 「打开小红书」
  - 「我已发布」

交互：

- 点击「复制文案」调用 Clipboard API。
- 点击「打开小红书」尝试跳转小红书 App Scheme 或 Web 地址；失败时提示用户手动打开。
- 点击「我已发布」进入奖励页。

### 6.5 奖励页

目标：给用户明确反馈，完成活动闭环。

内容：

- 发布成功提示。
- 奖励说明。
- 领取方式，例如「请向前台/教练出示此页面」。
- 活动规则提示。
- 二次分享或返回首页入口。

## 7. AI 内容生成要求

### 7.1 通义千问 API

- 使用前端环境变量配置 API Key。
- 推荐环境变量名：
  - `VITE_DASHSCOPE_API_KEY`
  - 或 `NEXT_PUBLIC_DASHSCOPE_API_KEY`
- 具体变量名随前端框架确定。
- 禁止将 API Key 硬编码到源码。
- 已暴露过的 API Key 建议轮换后再使用。

### 7.2 Prompt 目标

生成内容应适合深圳汇职驾校的小红书分享活动，语气真实、自然、轻商业化，避免过度营销。

输出 JSON，便于前端解析：

```json
{
  "title": "20字以内的小红书标题",
  "body": "适合用户直接发布的小红书正文",
  "tags": ["深圳驾校", "学车", "汇职驾校"],
  "materialType": "image",
  "style": "real_experience",
  "cta": "评论区或私信了解报名"
}
```

### 7.3 Prompt 建议

```text
你是小红书本地生活内容运营专家。请为“深圳汇职驾校”的分享活动生成一条用户可直接发布的小红书笔记。

要求：
1. 标题不超过 20 个中文字符。
2. 正文像真实学员分享，不要像硬广。
3. 内容突出深圳、本地学车、驾校服务、练车体验、报名咨询。
4. 正文分段清晰，语气自然，适合移动端阅读。
5. 生成 5-8 个小红书标签。
6. 返回严格 JSON，不要输出 Markdown。
```

## 8. 素材匹配要求

素材使用本地 Mock，不依赖远程资源。

建议目录：

```text
public/mock-assets/
  driving-school-front.jpg
  training-field.jpg
  coach-car.jpg
  student-practice.jpg
  license-success.jpg
  activity-poster.mp4
```

匹配逻辑：

- 如果 AI 返回 `materialType=image`，优先匹配图片。
- 如果 AI 返回 `materialType=video`，优先匹配视频。
- 如果标题或正文包含「练车」「训练场」，匹配训练场素材。
- 如果包含「拿证」「考试」，匹配拿证素材。
- 如果没有明确关键词，使用默认活动主视觉。

## 9. 发布模块扩展设计

前端应定义统一发布接口，MVP 阶段使用 Mock 实现。

```ts
export interface PublishPayload {
  title: string;
  body: string;
  tags: string[];
  assets: string[];
}

export interface PublishResult {
  success: boolean;
  mode: "mock" | "copy" | "deeplink" | "api";
  message?: string;
}

export interface Publisher {
  publish(payload: PublishPayload): Promise<PublishResult>;
}
```

MVP 可实现：

- `MockPublisher`：模拟发布成功。
- `CopyPublisher`：复制文案并引导用户手动发布。
- `DeeplinkPublisher`：尝试打开小红书。

后续可扩展：

- `XhsApiPublisher`：接入真实小红书发布能力。
- `ServerPublisher`：通过后端安全代理发布。

## 10. UI 与体验要求

- 手机端优先，设计宽度以 375px / 390px 为主要参考。
- 整体风格简洁、现代、高级、商业化。
- 避免复杂表单，用户默认只需点击一次「立即发布」。
- 底部主按钮常驻，适配 iOS 安全区。
- 动画自然克制，用于状态过渡和生成反馈。
- 视觉层级清晰，重要信息优先露出。
- 适配微信内置浏览器、Safari、Chrome。
- 所有按钮必须有加载、禁用、成功、失败状态。

## 11. 推荐技术方案

推荐使用：

- Vite + React + TypeScript
- CSS Modules / Tailwind CSS
- Framer Motion
- 本地 JSON 配置活动内容和素材规则

建议页面路由：

```text
/
/generating
/result
/publish
/reward
```

也可实现为单页状态机：

```ts
type AppStep = "intro" | "generating" | "result" | "publish" | "reward";
```

## 12. 验收标准

- 手机尺寸下页面可完整交互。
- 点击「立即发布」后能调用通义千问 API 并生成内容。
- API Key 从环境变量读取，不出现在源码中。
- 网络失败时有明确错误提示和重试按钮。
- 生成结果页能展示标题、正文、标签和本地素材。
- 用户能复制文案。
- 用户能进入模拟发布流程。
- 点击「我已发布」后能进入奖励页。
- 发布模块代码具有可替换接口，后续能接入真实发布能力。

## 13. 风险与注意事项

- 前端直连大模型 API 会暴露 API Key，即使使用环境变量也无法真正保护密钥。MVP 可接受，但正式上线建议增加后端代理。
- 小红书平台对 AI 托管运营和自动化发布存在治理要求，正式接入发布能力前必须确认合规边界。
- 自动生成内容应避免虚假承诺，例如「包过」「最快拿证」等高风险表达。
- 奖励领取规则必须在线下可执行，避免用户发布后无法核销。
